# C2 后台管理与系统可观测性

## 本次目标

推进 G#23-G#25，补齐 ADMIN 后台的用户权限管理、审查工具配置管理和系统可观测性入口。

## 完成内容

- G#19 审批事件恢复执行：
  - 新增 `ApprovalEventMessage`、`ApprovalEventPublisher` 和 `ApprovalEventConsumer`。
  - 审批通过、驳回、要求修改会在审批状态和历史写入成功后，通过 `approval.event.queue` 发布事件。
  - 事件消费者驱动审查任务从 `WAITING_APPROVAL` 恢复：审批通过后生成最终报告并进入 `COMPLETED`；驳回或要求修改后进入 `FAILED` 并记录原因。
  - 对重复审批事件做基础幂等保护，已完成或已失败任务不会重复推进。

- G#23 用户与角色管理：
  - 新增 `/admin/users` 用户列表、新建、更新和逻辑删除接口。
  - 新增 `/admin/roles` 与 `/admin/permissions` 查询接口。
  - 用户管理支持显示名、部门 ID、启停状态、密码设置/重置和角色分配。
  - 前端新增 `/admin/users` 用户与权限页面。

- G#24 审查工具配置管理：
  - 新增 `/admin/review-tools` 列表、新建、更新和逻辑删除接口。
  - 管理当前实际生效的 `review_tool_config` 配置，包括 `toolName`、`requiredPermission`、`riskLevel`、`approvalRequired` 和 `enabled`。
  - 前端新增 `/admin/tools` 审查工具配置页面。

- G#25 系统可观测性：
  - 在已有 Actuator/Micrometer 基础上新增 `/observability/summary` 汇总接口。
  - 返回合同、审查、待审批、启用工具、启用用户、审查状态分布、JVM 运行时和 Actuator 链接。
  - 前端新增 `/admin/observability` 系统可观测性页面。

## 说明

- 后台管理接口统一要求当前用户具备 `admin:manage` 权限。
- G#24 优先管理 `ToolPermissionGuard` 实际读取的 `review_tool_config` 表，确保页面配置会影响真实工具权限链路。

## 2026-05-31 MVP 闭环补齐

- G#12 Prompt 模板与版本管理：
  - 新增 `/admin/prompts` Prompt 模板列表、新建、更新、删除和预览接口。
  - 新增 `PromptTemplateService`，支持按场景解析启用模板，并渲染 `{{variable}}` 占位符。
  - 新增默认 Prompt 种子数据，覆盖条款抽取、风险分析、报告生成和知识库问答场景。
  - 前端新增 `/admin/prompts` Prompt 模板管理页面。

- G#20 审查报告模块：
  - `/reviews/{id}/report` 从仅返回审查与风险，扩展为返回合同基础信息、风险明细、审批记录、报告摘要和最终结论。
  - 前端报告页改为消费报告聚合接口，并提供跳转 Agent Trace 的入口。

- G#21 Agent Trace 模块：
  - `/reviews/{id}/trace` 扩展返回执行步骤、状态流转、LLM 调用日志、工具调用日志、RAG 检索日志和汇总指标。
  - RAG 检索写入 `tool_call_log`，便于 Trace 页面展示真实工具调用记录。
  - 前端新增 `/contracts/:id/trace` Agent Trace 详情页，按步骤、状态流转、LLM、工具和 RAG 分页查看。

- G#22 合规知识库问答模块：
  - 新增 `/knowledge-qa/ask`、`/knowledge-qa/history` 和反馈接口。
  - 问答流程先执行知识库检索，再用 `KNOWLEDGE_QA` Prompt 生成回答；无检索依据时明确拒答，不编造规则。
  - 问答结果写入 `qa_history`，保存引用片段和用户反馈。
  - 前端知识库问答页从“只展示检索结果”升级为问答、引用、历史和反馈闭环。
