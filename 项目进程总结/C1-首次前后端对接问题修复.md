# C1 首次前后端对接问题修复

## 背景

本次调试围绕前端首次连接真实后端后的合同上传、原文解析、删除、跨域、安全认证和长 ID 精度问题展开。目标是让“上传合同 -> 查看原文 -> 发起审查 -> 删除合同”等基础链路在浏览器环境中可稳定运行。

## 发现的问题

1. 上传 `.txt` 后前端详情页看不到合同原文。
   - 后端上传接口只保存文件和合同记录，没有在上传后立即解析并写入 `parsedText`。
   - `/contracts/{id}/original` 每次重新从磁盘读文件，未优先使用数据库中已有的 `parsedText`。
   - TXT 解析原先只按 UTF-8 读取，Windows 常见 GBK/ANSI 文本可能解析失败。

2. DOCX 文件无法解析。
   - 早期实现中 DOCX 明确返回“待后续解析器支持”，没有真实文本提取能力。

3. DELETE 请求先后出现 403。
   - 第一阶段 403 打在 `OPTIONS` 预检请求上，原因是跨域预检未显式放行。
   - 第二阶段 DELETE 本身看似 403，进一步定位发现有两类混淆：
     - 业务异常转发到 `/error` 后被 Spring Security 二次拦截，导致真实错误被空 403 掩盖。
     - 数据库雪花 ID 超过 JavaScript 安全整数范围，前端按 `number` 接收后精度丢失，DELETE 实际请求的 ID 与数据库 ID 不一致。

4. 前端 API 客户端对空响应体处理不完整。
   - DELETE 成功返回 `ApiResponse.ok(null)` 或空响应时，前端曾有 JSON 解析风险，后续已兼容空 body。

5. 删除后数据库仍保留记录。
   - 这是当前设计上的逻辑删除：`deleted=true`，列表过滤不展示，数据库保留审计与关联链路。

## 修复内容

- 合同上传后自动执行解析：
  - 解析成功写入 `parsedText`，合同状态改为 `PARSED`。
  - 解析失败写入 `metadata.parse_failure`，合同状态改为 `PARSE_FAILED`。

- 原文接口优化：
  - `/contracts/{id}/original` 优先返回数据库中的 `parsedText`。
  - `parsedText` 为空时才回退读取源文件。
  - 文件解析失败时返回明确错误，不再静默显示空原文。

- TXT/DOCX 解析增强：
  - TXT 支持 UTF-8 优先、GBK 兜底。
  - 引入 Apache POI `poi-ooxml`，支持 DOCX 正文抽取。
  - 新增 `ContractTextParserTest` 覆盖 UTF-8 TXT、GBK TXT、DOCX。

- CORS 与安全配置修复：
  - 禁用 CSRF，符合 JWT REST API 模式。
  - 显式放行 `OPTIONS /**`。
  - 增加 CORS 配置，允许本地前端来源、`Authorization` 请求头和 DELETE 方法。
  - 放行 `/error`，避免错误转发被安全链二次遮蔽。

- 全局异常处理：
  - 新增 `GlobalExceptionHandler`。
  - 业务错误返回统一 `ApiResponse.error(...)`。
  - not found -> 404，参数/状态错误 -> 400，数据约束冲突 -> 409，登录失败 -> 401。

- 长 ID 精度修复：
  - 后端新增 Jackson 配置，将 Java `Long/long` 序列化为字符串。
  - 前端将合同、审查、审批、风险、步骤、用户等业务 ID 类型改为 `string`。
  - 移除页面中对路由 ID 的 `Number(...)` 转换，避免雪花 ID 精度丢失。

## 验证情况

- 后端 `mvn test` 通过，25 个测试全绿。
- 前端 `npm.cmd run build` 通过。
- 使用真实 admin JWT 验证：
  - `GET /api/contracts` 可正常返回。
  - ID 精度修复后，前端应使用字符串 ID 发起 DELETE，避免误删不存在 ID。
